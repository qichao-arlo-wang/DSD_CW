\section{Scope and Objectives}
% =============================================================

This report covers Task 7 (7a/7b/7c) and Task 8 of the coursework.
For an input vector $X=\{x_0,x_1,\ldots,x_{N-1}\}$, the target computation is
\[
F(X)=\sum_{k=0}^{N-1} f(x_k),\quad f(x)=0.5x + x^3\cos\left(\frac{x-128}{128}\right)
\]
under single-precision floating-point representation.

% =============================================================
\section{Experimental Methodology}
% =============================================================

\subsection{Test Cases and Metrics}
The evaluation uses the three coursework-defined deterministic vectors and one final-assessment random vector. In explicit form, C1 is $X=\{0,5,10,\ldots,255\}$ with 52 samples, C2 is $X=\{0,\frac{1}{8},\frac{2}{8},\ldots,255\}$ with 2041 samples, and C3 is $X=\{0,\frac{1}{256},\frac{2}{256},\ldots,255\}$ with 65,281 samples. Case 4 is a pseudo-random vector with $N=2323$ generated using the required fixed seed, so software and reference-model comparisons are reproducible. Reported metrics are execution latency, FPGA resource usage (ALMs/LEs, DSPs, and memory bits), numerical error against a double-precision reference, and timing-closure margin at the 50\,MHz target.

% =============================================================
\section{Task 7a: CORDIC Analysis (10\%)}
% =============================================================

\subsection{Reference Model and Statistical Method}
Task 7a uses a MATLAB Monte Carlo model where random input $\theta \sim U[-1,1]$ is quantised to \texttt{single}, the reference output is computed as $y_{ref}=\cos(\theta)$ in single precision, and the CORDIC estimate $y_{cordic}$ is produced with fixed-point internal arithmetic. The error sample is defined as $e_k=(y_{cordic,k}-y_{ref,k})^2$, and the reported statistic is the MSE estimate $\hat{\mu}_e=\frac{1}{N}\sum_{k=1}^{N}e_k$ with the 95\% confidence interval $\hat{\mu}_e \pm 1.96\cdot s_e/\sqrt{N}$.
For all reported Task 7a results, we used $N=50{,}000$ samples and seed $=7$.

\subsection{Design Space Exploration}
We swept both required CORDIC parameters: all integer iteration counts $N_{iter}\in\{12,13,\dots,24\}$ and all integer fixed-point FRAC values from 20 to 34.
The total fixed-point width is set as:
\[
W = \mathrm{FRAC} + 2.
\]
The +2 integer/sign bits cover the CORDIC dynamic range for $\theta\in[-1,1]$ and $\cos(\theta)\in[-1,1]$.
The acceptance criterion is the coursework requirement:
\[
\text{CI}_{95,\text{upper}}(\text{MSE}) < 2.4\times10^{-11}.
\]

\subsection{Analytical Error Estimate ($2^{-i}$)}
For rotation-mode CORDIC, the residual angle after $i$ iterations is bounded by a term proportional to $2^{-i}$.
Using $|\Delta\cos(\theta)|\approx |\sin(\theta)\Delta\theta|\le|\Delta\theta|$, the absolute cosine error also follows an $O(2^{-i})$ trend; therefore squared error follows approximately $O(2^{-2i})$.
This estimate is not used as the final proof, but it is effective to prune the search space before Monte Carlo.

\subsection{Selected Configuration and Justification}
After including odd values, the first passing point in lexicographic order
(minimum iterations, then minimum FRAC) is:
\[
(N_{iter},W,\text{FRAC})=(17,23,21).
\]
Measured statistics for this minimum-passing point are MSE $=2.254\times10^{-11}$, 95\% CI $=[2.226, 2.282]\times10^{-11}$, and CI-upper-bound margin to the requirement of approximately $1.05\times$.
For implementation in Task 7b/7c we keep $(18,24,22)$ as a conservative choice:
with \texttt{ITER\_PER\_CYCLE}=3, both 17 and 18 iterations map to the same CORDIC cycle count (6 cycles), while $(18,24,22)$ has significantly larger numerical margin.

\subsection{Impact on Accuracy and Latency}
From the full sweep, FRAC$=20$ crosses the threshold at $N_{iter}=18$ (17 fails, 18 passes), while FRAC$\ge21$ already passes at $N_{iter}=17$. Increasing FRAC beyond this threshold region continues to reduce error but with diminishing practical benefit. Latency increases with iteration count for fixed iterations-per-cycle, and larger FRAC increases datapath cost, so both the minimum-passing point and a conservative implementation point are reported.

\section{Task 7b: CORDIC Architecture and Implementation (10\%)}
% =============================================================

\subsection{Data Availability and Missing Items (Declared Up Front)}
This section uses the Task 7a sweep statistics and plots, CORDIC functional simulation outcomes, software timing/error measurements for Step-2 and Step-3, and Quartus post-fit timing/resource results. For marking transparency, three limitations are declared up front: waveform capture with Questa/ModelSim was unavailable on this host, cycle-accurate vendor floating-point netlist simulation was unavailable in the RTL flow used here, and the optional dummy one-cycle custom-instruction hardware experiment was not implemented, so only its theoretical lower bound is reported.

\subsection{CORDIC Microarchitecture}
The cosine block uses rotation-mode fixed-point CORDIC with start/busy/done handshake:
\[
\begin{aligned}
x_{i+1} &= x_i - d_i(y_i \gg i),\\
y_{i+1} &= y_i + d_i(x_i \gg i),\\
z_{i+1} &= z_i - d_i\arctan(2^{-i}),\;\; d_i=\mathrm{sign}(z_i).
\end{aligned}
\]
Configuration used in hardware:
The hardware configuration is $N_{\text{iter}}=18$ with \texttt{ITER\_PER\_CYCLE}=3, which applies multiple CORDIC micro-iterations in each cycle and aligns with the full-mark direction. The selected fixed-point setting is FRAC$=22$ with a 28-bit CORDIC datapath width, intentionally conservative relative to the minimum-passing Task 7a point. The CORDIC compute stage therefore requires $\lceil 18/3 \rceil = 6$ micro-rotation cycles, and no LUT-based cosine or vendor cosine IP is used.

To justify the multi-iteration choice, we ran software-level Step-2 experiments with the same arithmetic flow and different \texttt{ITER\_PER\_CYCLE} settings:

\subsection{Verification and Timing}
Functional simulation of the standalone CORDIC block at representative points $\theta\in\{-1,-0.5,0,0.5,1\}$ passed with maximum absolute error $3.37\times10^{-6}$.
\textbf{Latency correction note:} an earlier configuration assumed 2-cycle FP add/sub/mul custom blocks. After integration testing, that setting produced noticeable numerical deviation because the FP IP pipeline latency was under-modeled at the wrapper level. The final implementation therefore uses 3-cycle latency for FP add/sub/mul.
Cycle-level isolated checks report 3 cycles each for FP\_MUL, FP\_ADD, and FP\_SUB under the final setting, and 9 cycles for the cosine custom-instruction wrapper (6 CORDIC compute cycles plus interface/control overhead). Post-fit timing for the Step-2 integration shows worst setup slack of 3.106\,ns, worst hold slack of 0.086\,ns, and Fmax of 59.19\,MHz on \texttt{sopc\_clk}, so the 50\,MHz target is met with positive margin.
No timing failures were observed for the reported selected architecture.

\subsection{Step 3: Final Single-CI Block}
Step-3 integrates the full inner function into one custom instruction:
\[
f(x)=0.5x + x^3\cos\left(\frac{x-128}{128}\right).
\]
The final block reuses one FP multiplier, one FP adder, and one CORDIC core under a single FSM, with local registers for intermediate values.

\subsection{Control and Data Flow}
In Step-2, the CPU writes operands to each custom instruction, blocks until completion, reads the result, and then issues the next call in sequence. In Step-3, the CPU issues one call per input $x$, and an internal FSM schedules all dependent operations until the final output is ready. Intermediate values are kept on-chip in local registers so no CPU-visible temporary transfers are required between arithmetic sub-steps. The resource-minimization rule is respected because each arithmetic sub-block is instantiated once and reused through time-multiplexed control.

\subsection{Expected vs Observed Latency}
Isolated-cycle estimates are much lower than Nios-integrated measurements. In Step-2, one sample evaluation of $f(x)$ performs five FP multiplications, one FP subtraction, one FP addition, and one cosine call, so the accelerator-only cycle estimate is
\[
T_{\text{Step-2,iso}} = 5\cdot 3 + 1\cdot 3 + 1\cdot 3 + 1\cdot 9 = 30\ \text{cycles/sample}.
\]
For C3 ($N=65{,}281$), this corresponds to about 39.2\,ms/run at 50\,MHz, while the measured Step-2 total latency is 2036.7\,ms/run. For Step-3, isolated single-CI latency is 21 cycles under the 3-cycle FP setting, which gives about 27.4\,ms/run idealized compute time for C3, while measured Step-3 total latency is 355.5\,ms/run.
The discrepancy is expected because Nios executes blocking custom-instruction calls and still incurs loop control, call/return, and memory traffic overhead. Step-3 reduces this overhead substantially by collapsing multiple calls into one CI call per sample.

\subsection{Standalone vs Nios-Integrated Throughput}
The standalone best-call interval from isolated Step-3 latency is 21 cycles at 50\,MHz, i.e. 420\,ns/call ($\sim$2.38\,M calls/s). The Nios-integrated effective interval from the C3 Step-3 run is 3555\,ms for 652,810 calls over 10 runs, i.e. 5.45\,$\mu$s/call ($\sim$183.6\,k calls/s, about 272 cycles/call equivalent). Therefore the Nios-integrated call interval is about 13$\times$ slower, dominated by software/interface overhead instead of arithmetic-core latency.

\subsection{Ideal Lower Bound (Dummy Combinational CI)}
The dummy one-cycle CI was not implemented in this submission; therefore this subsection reports the theoretical lower bound only.
If one $f(x)$ evaluation took exactly one clock cycle, C3 (65,281 samples) would require 65,281 cycles per run, i.e. 1.306\,ms at 50\,MHz. For the measured 10-run setup, this gives a lower bound of 13.06\,ms total, versus the measured Step-3 total of 3555\,ms. This gap confirms that the current bottleneck is system-level integration overhead rather than arithmetic datapath depth alone.

\section{Task 8: Throughput-Oriented Final Design (10\%)}
% =============================================================

\subsection{Architecture Evolution from Task 7}
Task 8 changes the optimization target from single-call latency to end-to-end throughput of the outer reduction $F(X)$. In Task 7 Step-3, each sample requires a custom-instruction call for $f(x)$ and software-side accumulation in the CPU loop. In Task 8, accumulation is moved into the custom-instruction path so each loop iteration can directly update the running sum through hardware. The CORDIC-based cosine computation is reused from Task 7, so numerical behavior is expected to remain aligned while reducing software orchestration work per sample.

The current integrated hardware/software run is based on a two-operand accumulate interface exposed by the BSP in this build, while a richer dual-input Task-8 block was also developed and validated in RTL simulation. This keeps the on-board integration simple, but it also means the measured hardware run in this section reflects the currently integrated CI contract rather than the full dual-input opcode interface explored in RTL.

\subsection{Final Task-8 Datapath and Scheduling}
The deployed Step-4 software flow issues one CI call per sample to update an accumulator state in hardware-oriented form. Conceptually, the datapath computes $f(x)$ and then combines it with the incoming accumulator in one CI transaction visible to software. This reduces explicit CPU-side arithmetic and lowers loop overhead compared with Step-3. The arithmetic core still reuses FP add/multiply and CORDIC cosine blocks derived from Task 7.

\subsection{Pipeline Utilisation and Throughput}
RTL cycle-level measurements of the dual-input Task-8 prototype show 28 cycles per pair-call, i.e. 14 cycles/sample effective, while the Task-7 single-$f(x)$ baseline is 17 cycles/sample. This corresponds to an isolated arithmetic improvement of about $17/14\approx1.21\times$ in cycles/sample. At 50\,MHz, the idealized arithmetic throughput bound is therefore about $50\text{M}/14\approx3.57$\,M samples/s.

From the current on-board Step-4 dataset, C3 uses 123.5\,ms/run for 65,281 samples, i.e. about 1.89\,$\mu$s/sample (approximately 0.53\,M samples/s). This is faster than Step-3 software-integrated throughput but still far below the isolated arithmetic bound, indicating substantial software/CI integration overhead remains. In addition, because current Step-4 outputs are functionally incorrect, throughput numbers should be interpreted as provisional integration diagnostics rather than final valid Task-8 performance.
