\section{Introduction and Methodology}
% =============================================================

This report covers Task 7 (7a/7b/7c) and Task 8 of the coursework.
For an input vector $X=\{x_0,x_1,\ldots,x_{N-1}\}$, the target computation is
\[
F(X)=\sum_{k=0}^{N-1} f(x_k),\quad f(x)=0.5x + x^3\cos\left(\frac{x-128}{128}\right)
\]
under single-precision floating-point representation.

To keep the report structure consistent with Report 2, this document also introduces an explicit baseline and then reports Task-7/Task-8 results as incremental evolution from that baseline.

\subsection{Baseline from Report 2}
The baseline used in this report is the Report-2 V6 configuration (FP custom add/sub/mul with \texttt{cosf} in software flow). This baseline is selected to provide a consistent pre-CORDIC reference point for Task-7 and Task-8 comparisons.

\section{Task 7a: CORDIC Analysis}
% =============================================================

\subsection{Reference Model and Sweep}
Task 7a uses a MATLAB Monte Carlo model with $\theta\sim U[-1,1]$ (\texttt{single}), compares CORDIC output to single-precision $\cos(\theta)$, and reports MSE with 95\% confidence bounds. The sweep uses all integer $N_{iter}\in[12,24]$ and FRAC$\in[20,34]$ with $W=\mathrm{FRAC}+2$.

\subsection{Selection Result}
The acceptance criterion is:
\[
\text{CI}_{95,\text{upper}}(\text{MSE}) < 2.4\times10^{-11}.
\]
After including odd values, the first passing point is
\[
(N_{iter},W,\text{FRAC})=(17,23,21).
\]
The implementation uses $(18,24,22)$ with \texttt{ITER\_PER\_CYCLE}=3 as a conservative point: it keeps the same CORDIC cycle count as 17 iterations under IPC=3 while providing larger error margin.

\section{Task 7b: CORDIC Architecture and Implementation}
% =============================================================

\subsection{CORDIC Microarchitecture}
The cosine block is rotation-mode fixed-point CORDIC with start/busy/done handshake. Final hardware parameters are $N_{iter}=18$, FRAC$=22$, width $W=24$, and \texttt{ITER\_PER\_CYCLE}=3, so the compute stage takes $\lceil18/3\rceil=6$ cycles. Increasing IPC improves cosine-block latency, but Step-2 end-to-end gain is limited by software orchestration overhead.

\subsection{Verification and Timing}
Standalone CORDIC simulation at $\theta\in\{-1,-0.5,0,0.5,1\}$ passed with maximum absolute error $3.37\times10^{-6}$. The final build uses 3-cycle FP add/sub/mul latency (instead of the earlier 2-cycle attempt) to avoid integration mismatch. Post-fit timing remains closed at 50\,MHz (setup slack 3.106\,ns, hold slack 0.086\,ns, Fmax 59.19\,MHz).

\section{Task 7c: Single Custom Instruction Integration}
% =============================================================

\subsection{Step-2 vs Step-3 Integration}
Step-2 uses separate FP\_MUL, FP\_ADD/SUB, and COS custom instructions with software scheduling. Step-3 integrates the full inner function into one custom instruction:
\[
f(x)=0.5x + x^3\cos\left(\frac{x-128}{128}\right).
\]
The Step-3 block reuses one FP multiplier, one FP adder, and one CORDIC core under an FSM, reducing CPU-visible calls from multiple operators per sample to one call per sample.

\subsection{Latency Interpretation}
Isolated-cycle estimates are lower than Nios-integrated measurements. In Step-2, one sample evaluation requires five FP multiplications, one subtraction, one addition, and one cosine call:
\[
T_{\text{Step-2,iso}} = 5\cdot 3 + 1\cdot 3 + 1\cdot 3 + 1\cdot 9 = 30\ \text{cycles/sample}.
\]
For C3, this predicts about 39.2\,ms/run at 50\,MHz versus measured 2036.7\,ms/run. Step-3 isolated latency is 21 cycles (about 27.4\,ms/run idealized for C3) versus measured 220.0\,ms/run. The remaining gap is mainly software/interface overhead, but Step-3 significantly reduces it by collapsing call granularity.

\subsection{Ideal Lower Bound (Dummy Combinational CI)}
The dummy one-cycle CI was not implemented in this submission; therefore this subsection reports the theoretical lower bound only.
If one $f(x)$ evaluation took exactly one clock cycle, C3 (65,281 samples) would require 65,281 cycles per run, i.e. 1.306\,ms at 50\,MHz. For the measured 10-run setup, this gives a lower bound of 13.06\,ms total, versus the measured Step-3 total of 2200\,ms. This gap confirms that the current bottleneck is system-level integration overhead rather than arithmetic datapath depth alone.

\subsection{Key Trade-offs}
The baseline comparison clarifies the core trade-off. Relative to the Report-2 V6 baseline, Task-7 Step-2 already improves latency because the CORDIC-based cosine path is cheaper than the software \texttt{cosf} bottleneck in V6. Task-7 Step-3 then collapses the inner arithmetic sequence into one custom instruction call per sample and gains a much larger system-level improvement, reaching about $19.91\times$ speedup on C3 relative to baseline while preserving correctness within the measured error bound. This confirms that both algorithm choice (CORDIC replacing \texttt{cosf}) and call granularity are decisive in the Nios-integrated setting.

Task-8 Step-4 further reduces reported run time, but current hardware/software integration still shows functional mismatch in the measured dataset (relative error 1.0). Therefore its timing numbers are best interpreted as integration-stage throughput indicators, not final valid acceleration results. In resource terms, the Task-7/Task-8 builds remain in a similar area class and continue to close timing at 50\,MHz, so the remaining bottleneck is primarily interface/control behavior rather than FPGA timing closure.

\section{Conclusions}
% =============================================================

Using the Report-2 V6 result as baseline, the final validated Task-7 architecture shows that the most effective gain comes from combining CORDIC-based cosine with reduced CPU-visible call count. The CORDIC precision study in Task 7a identifies a minimum passing point and a conservative implementation point, and the implemented Task-7 Step-3 single-CI flow reaches about $19.91\times$ C3 speedup versus baseline while maintaining acceptable numerical accuracy and 50\,MHz timing closure.

Task-8 measurements indicate additional latency reduction potential beyond Task-7 Step-3, but the current on-board dataset is not yet functionally correct, so those gains cannot be claimed as final performance. The main remaining engineering target is to preserve Task-8 throughput behavior while restoring functional correctness under the integrated CI interface. Once correctness is closed, the same baseline framework used here can directly produce a final validated speedup figure for submission.
